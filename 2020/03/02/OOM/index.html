内存溢出-OOM

JVM OOM

OOM架构

java.lang.StackOverflowError

此错误为栈内存溢出，出现原因是深度的方法调用。

public class StackOverflowDemo {
    public static void main(String[] args){
        testDemo();
    }
    private static void testDemo(){
        testDemo();
    }
}
//Exception in thread "main" java.lang.StackOverflowError
//    at StackOverflowDemo.testDemo(StackOverflowDemo.java:6)
//    at StackOverflowDemo.testDemo(StackOverflowDemo.java:6)
//    at StackOverflowDemo.testDemo(StackOverflowDemo.java:6)

java.lang.OutOfMemoryError

Java heap space

由于对象太多导致内存溢出。

public class JavaHeapSpaceDemo {
    public static void main(String[] args) {
        byte[] bytes =new byte[80 * 1024 * 1024];//80MB
    }
}
//修改运行时虚拟机参数：Run ➞ Edit Configurations ➞ VM options：-Xms10m -Xmx10m
//Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
//    at JavaHeapSpaceDemo.main(JavaHeapSpaceDemo.java:3)

GC overhead limit exceeded

由于GC回收时间过长导致内存溢出。

过长的定义是：超过98%的时间用来做GC并且回收了不到2%的堆内存，连续多次GC都只回收了不到2%内存的极端情况下才会出现此错误。假如不抛出GC overhead limit exceeded会出现什么情况呢？那就是GC清理出的那点内存很快会再次被填满，迫使再次执行GC，恶性循环。CPU使用率一直是100%但是GC却没有明显效果。

public class GCOverHeadDemo {
    public static void main(String[] args){
        int i = 0;
        List<String> list = new ArrayList<>();
        try {
            while(true){
                list.add(String.valueOf(++i).intern());
            }
        }catch (Throwable e){
            e.printStackTrace();
            throw e;
        }
    }
}
//修改运行时虚拟机参数：-Xms10m -Xmx10m -XX:+PrintGCDetails -XX:MaxDirectMemorySize=5m
//java.lang.OutOfMemoryError: GC overhead limit exceeded
//    at java.lang.Integer.toString(Integer.java:401)
//    at java.lang.String.valueOf(String.java:3099)
//    at GCOverHeadDemo.main(GCOverHeadDemo.java:10)

Direct buffer memory

由于本地内存不足导致内存溢出。

写NIO程序经常使用ByteBuffer对象来读取或者写入数据，这是一种基于通道(Channel)与缓冲区(Buffer)的I/O方式。它可以使用Native函数库直接分配堆外内存，然后通过一个存储在Java堆里面的DirectByteBuffer对象作为这块内存的引用进行操作。这样能在一些场景中显著提高性能，避免了在Java堆和Native堆中来回复制数据。

  • ByteBuffer.allocate(capability)：这种方式是分配JVM堆内存，属于GC管辖范围，由于需要拷贝所以速度相对较慢。

  • ByteBuffer.allocateDirect(capability)：这种方式是分配系统本地内存，不属于GC管辖范围，不需要内存拷贝速度快。

但如果不断分配本地内存，堆内存很少使用，那么JVM就不需要执行GC，DirectByteBuffer对象们就不会被回收。这时候虽然堆内存充足，但是本地内存已经使用光了，再次尝试分配本地内存就会出现OutOfMemoryError。

public class DirectBufferMemoryDemo {
    public static void main(String[] args) {
        System.out.println("配置的最大直接内存："+(sun.misc.VM.maxDirectMemory()/(double)1024/1024)+"MB");
        ByteBuffer bb = ByteBuffer.allocateDirect(6*1024*1024);//6MB
    }
}
//修改运行时虚拟机参数：-Xms10m -Xmx10m -XX:+PrintGCDetails -XX:MaxDirectMemorySize=5m
//配置的最大直接内存：5.0MB
//Exception in thread "main" java.lang.OutOfMemoryError: Direct buffer memory
//    at java.nio.Bits.reserveMemory(Bits.java:694)
//    at java.nio.DirectByteBuffer.<init>(DirectByteBuffer.java:123)
//    at java.nio.ByteBuffer.allocateDirect(ByteBuffer.java:311)
//    at DirectBufferMemoryDemo.main(DirectBufferMemoryDemo.java:6)

unable to create newnative thread

由于应用程序创建太多线程超过系统承载极限导致内存溢出。

这个错误在高并发请求服务器时比较常见，linux系统默认允许单个进程可以创建1024个线程（1024是理论数据，一般到900多就会抛异常），如果超过这个数量就会报OutOfMemoryError:unable to create newnative thread。

public class UnableCreateNewThreadDemo {
    public static void main(String[] args){
        for (int i=1;;i++){//注意for循环中没有配置范围
            System.out.println(i);//linux下创建940个线程后抛出异常
            new Thread(()->{
                try {
                    Thread.sleep(Integer.MAX_VALUE);
                }catch (InterruptedException e){
                    e.printStackTrace();
                }
            },""+i).start();
        }
    }
}
//Exception in thread "main" java.lang.OutOfMemoryError: unable to create new native thread
//    at java.lang.Thread.start0(Native Method)
//    at java.lang.Thread.start(Thread.java:717)
//    at UnableCreateNewThreadDemo.main(UnableCreateNewThreadDemo.java:11)

解决方案无非两种：降低应用程序创建线程的数量、修改linux系统的服务器配置扩大限制。

在linux的普通用户下：$ ulimit -u

得到的返回结果是：$ 1024

$ vim /etc/security/limits.d/90-nproc.conf

*        soft        nproc        1024
root     soft        nproc        unlimited

如必要扩大限制，可以在root用户下添加普通用户并设置线程数量，不要直接修改 *

Metaspace

由于元空间满了导致内存溢出。

Java8以后使用Metaspace来替代永久代。查看元空间的初始大小的方式如下：

➜ demo java -XX:+PrintFlagsInitial

永久代（元空间）存放了以下信息：

  • 虚拟机加载的类信息

  • 常量池

  • 静态变量

  • 即时编译后的代码

Metaspace元空间与永久代最大的区别在于：Metaspace并不在虚拟机内存中，而是使用本地内存。

public class MetaSpaceDemo {
    static class OOMTest {} //静态内部类
    public static void main(String[] args) {
        int i = 0;
        try {
            while (true){
                i++;
                Enhancer enhancer = new Enhancer();
                enhancer.setSuperclass(OOMTest.class);
                enhancer.setUseCache(false);
                enhancer.setCallback(new MethodInterceptor() {
                    @Override
                    public Object intercept(Object o, Method method, Object[] objects, MethodProxy methodProxy) throws Throwable {
                        return methodProxy.invokeSuper(o,args);
                    }
                });
                enhancer.create();
            }
        }catch (Throwable e){
            System.out.println(i+"次后发生异常");
            e.printStackTrace();
        }
    }
}
//修改运行时虚拟机参数：-XX:MetaspaceSize=8m -XX:MaxMetaspaceSize=8m
//362次后发生异常
//Caused by: java.lang.OutOfMemoryError: Metaspace
//    at java.lang.ClassLoader.defineClass1(Native Method)
//    at java.lang.ClassLoader.defineClass(ClassLoader.java:763)
