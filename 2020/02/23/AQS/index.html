AQS（AbstractQueuedSynchronizer）

private transient volatile Node head;//队首

private transient Thread exclusiveOwnerThread;//持有锁的线程

Node类的设计

公平锁｜非公平锁

final void lock() {
    acquire(1);//加锁成功之后改变的值
}

final void lock() {
    if (compareAndSetState(0, 1))
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1);
}

公平锁的上锁必须判断自己需不需要排队；而非公平锁是直接进行CAS修改计数器看能不能加锁成功；如果加锁不成功则乖乖排队(调用acquire)；所以不管公平还是不公平；只要进到了AQS队列当中那么就会排队。

acquire()方法源码分析

public final void acquire(int arg) {
    if (!tryAcquire(arg) && //①
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))//②③
        selfInterrupt();
}
//①tryAcquire()：尝试加锁，如果加锁失败就会调用acquireQueued()去排队，加锁成功就不会。
//②addWaiter()：判断队列是否需要初始化，并把当前线程节点放入阻塞队列。
//③acquireQueued()：当前线程如果排在第一位会请求获取锁，否则入队睡眠阻塞，后续线程修改睡眠状态，等待被唤醒。

①tryAcquire()

protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();//获取当前线程
    int c = getState();//获取lock对象的状态c=0自由态，如果被上锁则为1，大于1是重入
    if (c == 0) {
        if (!hasQueuedPredecessors() &&//这里涉及到的情况比较多，下面详细讨论
            compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);//设置持有锁的线程是自己
            return true;
        }
    }
    //如果C不等于0，判断当前线程是否为拥有锁的线程，如果不是则直接返回false，加锁失败。
    //如果是拥有锁的线程，则状态+1代表重入次数+1，返回true加锁成功。
    else if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;
        if (nextc < 0)
            throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    return false;
}

hasQueuedPredecessors()

public final boolean hasQueuedPredecessors() {
    Node t = tail;
    Node h = head;
    Node s;
    return h != t && //这句代码涉及到的情况比较多，下面详细解释
        ((s = h.next) == null || s.thread != Thread.currentThread());
}

第一种情况：队列没有初始化，head和taill都为null，所以h!=t不成立，后面短路不执行，直接返回false。
而又因为前面取反：if (!hasQueuedPredecessors()…)，所以不需要排队，直接去CAS，改变计数器上锁。

第二种情况：队列已经初始化了，但是有且只有一个数据，即h==t且s==null，表达式后面不执行，同上返回false不排队。什么时候会出现这种情况：当排队的队尾拿到锁的时候，此时队列中有且只有一个节点，它会把自己设置为h。因为这个线程已经拿到锁了，它不参与排队，所以需要设置为头节点。

第三种情况：队列已经初始化了，有大于1的多个节点，h!=t成立，s=null不成立，所以还要运行 s.thread != Thread.currentThread()，这里又分为两种情况：

  1. s.thread != Thread.currentThread()为true，即排队的第一个线程不是当前线程，表达式整体返回true，需要排队。

  2. s.thread != Thread.currentThread()为false，即排队的第一个线程就是当前线程，表达式整体返回false，不需要排队。

    不需要排队则调用 compareAndSetState(0, acquires) 去改变计数器尝试上锁，这里还分为两种情况：

    1. 第一种情况是加锁成功，即持有锁的线程释放了锁，继续执行setExclusiveOwnerThread(current)，返回true。
    2. 第二种情况是加锁失败，即持有锁的线程还没有释放锁，直接返回false。

②addWaiter()

private Node addWaiter(Node mode) {
    //因为AQS队列中的元素为Node，所以把当前线程封装为一个Node对象
    Node node = new Node(Thread.currentThread(), mode);
    Node pred = tail;//把队尾赋值给临时节点pred
    if (pred != null) {//队列不为空的流程很简单，直接修改指针把节点放入队列
        node.prev = pred;//当前线程节点指向原来的队尾节点
        if (compareAndSetTail(pred, node)) {//赋值操作，即队尾⇨当前节点
            pred.next = node;//原来队尾节点的指针指向当前节点
            return node;
        }
    }
    enq(node);//队列为空，初始化队列并进行入队操作
    return node;
}

enq()

private Node enq(final Node node) {
    //死循环    
    for (;;) {
        Node t = tail;//队尾赋值给临时节点t
        //第一次循环（初始化队列）
        if (t == null) {//队列为空，需要初始化队列
            if (compareAndSetHead(new Node()))//实例化一个属性都为空的Node对象
                tail = head;//队列中只有一个对象，头尾都指向它
        //第二次循环（当前线程节点入队）
        } else {
            node.prev = t;//⑴
            if (compareAndSetTail(t, node)) {//⑵
                t.next = node;//⑶
                return t;//上面123代表的是维护链表，即当前线程节点的入队操作
            }
        }
    }
}

③acquireQueued()

final boolean acquireQueued(final Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            final Node p = node.predecessor();//把当前节点的上一个节点赋值给P
            //如果当前节点的前置节点是头节点，那么它会尝试去获取锁--自旋（第一个排队的）
            if (p == head && tryAcquire(arg)) {//意图是尽量不睡眠，多询问一次
                setHead(node);
                p.next = null; // help GC
                failed = false;
                return interrupted;
            }
            //自旋获取锁失败后是否需要睡眠
            if (shouldParkAfterFailedAcquire(p, node) &&//给上一个节点修改状态
                parkAndCheckInterrupt())//②当前线程调用park()阻塞
                interrupted = true;
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}

shouldParkAfterFailedAcquire()

private static boolean shouldParkAfterFailedAcquire(Node pred, Node node) {
    int ws = pred.waitStatus;//获取前置线程节点的状态
    if (ws == Node.SIGNAL)//判断前置线程的状态是否为-1（已睡）
        return true;//跳出方法后运行parkAndCheckInterrupt()当前线程睡眠
    if (ws > 0) {
        do {
            node.prev = pred = pred.prev;
        } while (pred.waitStatus > 0);
        pred.next = node;
    } else {//把前置节点的状态修改为-1，证明前置线程已睡
        compareAndSetWaitStatus(pred, ws, Node.SIGNAL);
    }
    return false;//跳出方法后短路，再次自旋
}

synchronized和lock的区别

  1. 原始构成

    synchronized是关键字，属于JVM层面。底层是通过monitor对象来完成，wait/notify方法也依赖moniter对象并且只有在同步块或者同步方法中才能使用wait/notity等方法。monitorenter｜monitorexit｜monitorexit 正常退出和异常退出，不会死锁。

    lock是具体的类java.util.concurrent.locks.lock属于API层面。

  2. 使用方法

    synchronized不需要用户去手动释放锁，当synchronized代码执行完后系统会自动让线程释放对锁的占用。

    ReentrantLock需要用户去手动释放锁，如果没有主动释放锁就有可能导致死锁。需要lock()和unlock()配合try/finally来完成。

  3. 等待是否可中断

    synchronized不可中断，除非抛出异常或者正常运行完成。

    ReentrantLock可中断，相比synchronized更加灵活，有两种方式可以中断：

    • 设置超时方法tryLock(long timeout,TimeUnit unit)

    • lockInterRuptibly()放代码块中，调用interrupt()方法可中断。

  4. 加锁是否公平

    synchronized是非公平锁。

    ReentrantLock两者都可以，默认非公平锁。构造方法可以传入boolean值，true为公平锁，false为非公平锁。

  5. 是否可绑定多个条件-Condition

    synchronized不可以，要么随机唤醒一个线程，要么全部唤醒。

    ReentrantLock可以实现分组唤醒，达到精确唤醒。

扩展：jdk1.6以前，如果线程是交替执行的话，Synchronized会调用OS函数，影响性能。ReentrantLock的交替执行是在jdk级别解决的，性能很快。如果是竞争执行的话会进入队列调用park()方法。1.8以后区别很小，ReentrantLock的API更丰富一些。
